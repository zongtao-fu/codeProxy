# 可维护性优化计划（Optimization Plan）

本文件用于记录“面向未来的改进计划”，与 `docs/evolution.md` 不同：

- `docs/evolution.md`：记录已经发生的结构性变更与关键决策（why/影响/回滚）。
- `docs/optimization-plan.md`：记录还没做、但值得做的优化方向与里程碑（目标/方案/验证）。

## 什么时候写

- 发现重复劳动、长期技术债、架构边界混乱、性能瓶颈、测试薄弱、CI 反复失败时。
- 需要做中大型重构，但又希望可追溯、可拆分、可回滚时。

## 记录原则

- 每条计划都要可交付：拆成 1–2 天可完成的小里程碑，避免“写了但永远做不完”。
- 目标尽量可量化：例如“构建时间降低 30%”“把 1k+ 行文件拆成 3 个模块并保持行为不变”。
- 必须写清楚验证方式：至少包含“如何验证没有引入回归”。

## 条目模板（可复制）

```
## YYYY-MM-DD · 优化主题

### 现状与痛点

### 目标（可量化）

### 非目标（明确不做什么）

### 方案候选与对比（2–3 个即可）
- 方案 A：
- 方案 B：

### 推荐方案与理由

### 分阶段里程碑（建议按 1–2 天拆分）
- Milestone 1：
- Milestone 2：

### 风险与回滚策略

### 验证清单（最小闭环）
- Lint：
- 格式化：
- 构建：
- 测试（如有）：
```
