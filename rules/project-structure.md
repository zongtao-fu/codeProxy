# 项目结构规范（Project Structure Rules）

## Purpose

- 统一目录职责与依赖方向，降低耦合与维护成本。
- 在“最小模块化”前提下，保持边界清晰、可持续演进。

## When to Read

- 新增/移动文件或重构模块边界时。
- 不确定某段代码应该放在哪个目录时。

## Rules

### 1) 用目录表达耦合度（强制）

- 一句话口径：**越靠近模块核心，耦合越强；越远离模块核心，复用越高、耦合越低。**
- **高相关必须靠近**：经常一起改、强依赖、同一变化原因的文件应放在同一模块目录内相邻位置。
- **低相关必须远离**：跨模块复用的能力要么下沉到更稳定的公共层（如 `packages/`、`lib/`），要么通过清晰入口导出隔离。
- **远距离导入是边界警报**：出现跨模块深路径导入通常意味着边界划分不合理；优先重划边界/下沉公共层，而不是继续堆导入路径。

### 2) 依赖方向（强约束）

- 公共层（如 `packages/*` / `lib/*`）不得反向依赖应用/业务层。
- 模块间依赖优先通过稳定入口（如 `index.ts` / `__init__.py` / `mod.rs`）导出，避免跨层深路径导入。

### 3) 单文件复杂度控制（建议）

- 单文件过长/职责过多时优先拆分（抽子模块/抽组件/抽 hooks/抽 utils）。
- 优先拆“变化原因不同”的部分，避免为了“复用”过度抽象。
