# 工程质量规范（Quality Rules）

## Purpose

- 保证代码可维护、可扩展、可验证。
- 让“最小模块化”不等于“到处堆代码”，而是可演进的边界管理。

## When to Read

- 设计/重构架构边界、编写核心逻辑、或出现性能/稳定性问题时。

## Rules

### 1) 架构原则

- 高内聚低耦合：相关变化原因放在一起；跨模块依赖通过最小公共 API 收敛。
- DRY / KISS：能不抽象就不抽象；一旦抽象就要能复用并减少复杂度。
- 单向依赖：依赖方向必须稳定（见 `rules/project-structure.md`）。

### 2) 代码质量

- 命名清晰、语义明确；避免缩写堆叠与“一眼看不懂”的泛化名称。
- 关键逻辑注释说明“为什么这样做”，并能指导后续维护者。
- 不保留废弃代码与无价值兜底；失败场景尽量上移到统一边界处理。
- 任何改动必须保持可编译/可运行；禁止占位实现与把问题留给后续。

### 3) 性能与体验

- 避免在热点路径中做昂贵计算；必要时做缓存、memo 或分层。
- UI/交互类应用：避免在 resize/drag 等高频事件里触发昂贵重绘。

### 4) 测试与验证

- 优先通过可运行的构建/检查命令形成最小验证闭环（见 `rules/workflow.md`）。
- 可单测的纯函数/工具尽量保持无副作用、输入输出明确。
