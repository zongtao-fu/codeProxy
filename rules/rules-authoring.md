# 规范维护（Rules Authoring）

## Purpose

- 保证规范是“可执行、可演进、低噪声”的约束集合，而不是堆砌说明书。

## When to Read

- 需要新增/调整规则、或你发现现有规则不再适用时。

## Rules

### 1) 根 AGENTS.md 的定位（硬性）

- `AGENTS.md` 只做**入口索引**与**按需读取**引导。
- 具体细则必须拆到 `rules/*`，避免把所有规则堆在根文件里。

### 2) 规则拆分原则

- 一条规则应当能回答“遇到什么情况 → 该读哪份规则 → 怎么做 → 怎么验证”。
- 规则越具体越好，但必须与仓库强相关，避免通用常识堆砌。
- 若规则冲突：以更严格/更高优先级为准，并在相关文件中补充说明。

### 3) 演进记录（必须）

- 所有结构性变更（目录调整、依赖方向变更、工具链切换、重要架构决策）必须记录到 `docs/evolution.md`：
  - 时间（YYYY-MM-DD）
  - 变更内容
  - 关键决策与原因（why）
  - 影响范围与回滚要点（如适用）

### 4) 索引联动（必须）

- 新增/删除规则文件后，必须同步更新：
  - `AGENTS.md`（规则索引与任务类型映射）
  - 必要时更新 `shrimp-rules.md`（硬性约束变更时）

### 5) 项目路径变更联动（必须）

当发生“项目路径变更”时（新增/移动/重命名目录或文件、调整导出入口、更新别名路径、改变关键配置位置等），必须同步完成：

- 更新 `AGENTS.md`：确保索引、任务类型映射与关键路径描述仍然准确；
- 更新 `rules/project-structure.md`：若结构/边界/依赖方向有变化，必须反映到规范；
- 更新 `README.md`：若目录说明或启动/构建路径受到影响；
- 更新 `docs/evolution.md`：若属于结构性变更，必须记录原因与影响。
