# 流程规范（Workflow Rules）

## Purpose

- 统一从需求到落地的执行流程，降低返工与遗漏风险。
- 强制形成“实现 → 自检 → 验证”的闭环。

## When to Read

- 任何需要写代码/改结构/升级依赖的任务开始前。
- 任务包含多个步骤、或你不确定风险边界时。

## Workflow Overview

```
明确目标与约束 → 按需读取规则 →（复杂改造：建立会话文档）→ 阅读现有代码 → 方案对比与选择 → 实现 → 自检 → 验证 → 记录演进
```

## 复杂改造：会话文档落盘（分级）

本仓库不要求每次小改动都建立会话目录；仅在“复杂改造/大范围重构”时建议/要求落盘会话文档，以保证可追溯、可回滚与协作一致性。

### A) 不需要建立会话目录（轻量改动）

满足全部条件即可不建：

- 修改范围很小（通常 1–2 个文件、或一个小功能点），且都在同一模块内
- 无路径/结构变更（无新增/移动/重命名目录或文件、无导出入口/别名调整）
- 无依赖升级/新增
- 不触发“危险操作确认机制”的任何条目
- 可在一次提交完成，且回滚成本低

此类任务仍需做到：

- 在 PR/commit 中把“做了什么 + 为什么”写清楚
- 形成最小验证闭环（至少）：
  - Lint：`bun run lint`
  - 格式化：`bun run format`
  - 格式化检查：`bunx oxfmt . --check`
  - 构建：`bun run build`
  - （可选）测试：暂无（当前仓库未配置测试脚本/用例）

### B) 建议建立会话目录（中等改动）

满足任一条件建议建立：

- 需要 2 个及以上提交才能完成
- 修改文件较多（例如 3–10 个），或需要跨多个子模块联动（但不涉及路径变更）
- 引入新的对外导出/公共 API（需要说明边界与使用方式）

建议落盘（可按项目调整目录）：

- `.sisyphus/sessions/YYYY-MM-DD--topic/plan/plan.md`（至少）
- 尽量补齐 `changes.md` 与 `test-report.md`（推荐）

### C) 必须建立会话目录（大范围改造/高风险）

满足任一条件强制建立：

- 路径变更（新增/移动/重命名目录或文件、调整关键导出入口、更新别名等）
- 跨多个模块/包的结构性调整
- 依赖升级/版本策略调整（尤其是大版本升级）
- 触发“危险操作确认机制”任何条目

强制要求（最小闭环三件套）：

1. 在开始实现前，在 `.sisyphus/sessions/` 下新建本次会话目录（建议命名：`YYYY-MM-DD--topic`）。
2. 在会话目录下创建 `plan/`（文本类文档放这里）。
3. `plan/` 目录至少包含：
   - `plan.md`：任务计划（目标/非目标/清单/风险/回滚/验证）
   - `changes.md`：变更说明（做了什么/影响什么/改了哪些文件/为什么）
   - `test-report.md`：验证与测试报告（必须来自本次真实执行）

## 交付自检清单（强制）

### 1) 优雅性与复用（先做）

- 是否已有可复用组件/Hook/工具函数？优先复用，避免重复实现。
- 是否把变化点收敛到最合适的边界（高聚合）：对外暴露最小 API，内部细节自包含。
- 是否保持依赖方向正确（见 `rules/project-structure.md`）：避免跨模块直接耦合与深层相对路径引用。
- 文件是否需要继续拆分：单文件是否过长；是否应该抽子组件/抽 hooks/models/utils。

### 2) 注释与冗余（再做）

- 注释是否解释“为什么（why）”，而不是重复代码在做什么（what）。
- 是否存在冗余逻辑、重复兜底、过度防御；能用更清晰结构表达就应重写。

### 3) Review 自检（必须）

- 无用兜底/死代码：未使用变量、不可达分支、重复状态、重复组件/重复工具函数。
- 风险逻辑：危险操作是否有二次确认与明确提示（如丢弃/回滚/覆盖写入）。
- 输入与安全：路径拼接/相对路径解析是否防穿越；外部命令调用是否可控、错误是否可读。

### 4) 语法/类型/编译（必须）

- 确保类型检查/编译通过（按项目工具链）。

### 5) 测试与验证（必须）

- 最少跑通（按本仓库现状自动识别）：
  - Lint：`bun run lint`
  - 格式化：`bun run format`
  - 格式化检查：`bunx oxfmt . --check`
  - 构建：`bun run build`
  - （可选）测试：暂无（当前仓库未配置测试脚本/用例）
- 交付/合并前验证（推荐）：
  - 一键验证：`bun run check`

## 危险操作确认机制（必须先说明风险）

以下操作属于高风险，执行前必须先说明影响范围与回滚策略：

- 删除文件/目录或大规模重构目录结构
- 数据库迁移、数据回填、不可逆数据修复
- 依赖大版本升级或工具链切换
- 不可逆命令（如 `git reset --hard`、强制 push）

确认提示模板：

```
【危险操作检测】
操作类型：xxx
影响范围：xxx
风险与后果：xxx
回滚策略：xxx
验证方式：xxx
```

### 本项目约定：复杂任务必须维护 Todolist

- 当任务属于“对齐/迁移/重构/多步骤交付”（通常需要多文件联动或多轮验证）时，必须在 `.helloagents/plan/` 下创建本次任务目录，并维护 `tasks.md` 作为 todolist。
- 实施过程中每完成一个里程碑（例如：完成一个子功能、完成一次验证、修复一次阻断问题），都要**及时更新**该 `tasks.md`：
  - 勾选已完成事项
  - 更新 `<!-- LIVE_STATUS_BEGIN -->` 区块里的“状态/进度/更新时间/当前”
- 交付前必须保证：todolist 与实际代码变更一致，避免“已完成但未记录 / 已记录但未完成”。
